# Project Requirement Document
## Platform: Developer Q&A Community

---

## 1. Project Overview

You are required to build a **community-driven Developer Q&A Platform** ŌĆö similar to Stack Overflow ŌĆö where developers can post technical questions, share knowledge, help peers, and engage in meaningful discussions. The platform must be intelligent, secure, and fair for all users.

### Core Objectives

- Build a fully functional Q&A platform with user-generated content.
- Implement role-based access control (Guest, Registered User, Admin).
- Integrate an **AI model** to automatically detect and handle spam/abusive content.
- Apply smart **rate limiting** to prevent abuse and server overload.
- Build a comprehensive **Admin Dashboard** for platform monitoring and management.

## 4. Functional Requirements

---

### 4.1 Authentication & Authorization

#### 4.1.1 Registration

- Users must be able to register with the following fields:
 - Full Name
 - Username (unique)
 - Email Address (unique)
 - Password (must be hashed before storing ŌĆö use bcrypt or equivalent)
 - Profile Picture (optional)
- Email format validation is required.
- Password must meet minimum security criteria:
 - At least 8 characters
 - At least 1 uppercase letter
 - At least 1 number

#### 4.1.2 Login

- Users must be able to log in using **email + password**.
- On successful login, issue a **JWT token** (or equivalent session token).
- Token must have an expiry time (recommended: 7 days for remember-me, 1 hour for default).

#### 4.1.3 Login Rate Limiting 

> This is a critical security requirement.

- Users are allowed a maximum of **10 login attempts within any 5-minute window**.
- On the **10th failed attempt**, the login endpoint must be **blocked for 5 minutes** for that specific IP address or user account (or both).
- After the 5-minute block expires, the attempt counter resets.
- A clear error message must be shown to the user when blocked:
 > _"Too many failed login attempts. Please try again after 5 minutes."_
- Blocking must be tracked per **IP address** and optionally per **user account**.

#### 4.1.4 Logout

- Provide a logout option that clears the user's session/token.

#### 4.1.5 Access Control Rules

| Feature | Guest | Registered User | Admin |
|---|---|---|---|
| Read Posts | Yes | Yes | Yes |
| Search Posts | Yes | Yes | Yes |
| Write / Post Question | No | Yes | Yes |
| Reply to a Post | No | Yes | Yes |
| Upload Screenshots | No | Yes | Yes |
| Receive Notifications | No | Yes | Yes |
| Access Admin Panel | No | No | Yes |

### 4.2 Post Management

#### 4.2.1 Creating a Post

- A **logged-in user** can create a new post (question/query).
- Required fields:
 - **Title** ŌĆö A short, descriptive title of the question (max 200 characters)
 - **Description** ŌĆö Detailed explanation of the problem (supports rich text or markdown)
 - **Tags** ŌĆö At least 1 tag must be selected or created (e.g., `javascript`, `react`, `sql`)
- Optional fields:
 - **Screenshot / Image** ŌĆö User can upload one or more screenshots to support the question.
 - Accepted formats: JPG, PNG, WEBP, GIF
 - Max file size per image: 5 MB
 - Max 3 images per post
- Posts are visible to **everyone** (including guests) immediately after submission (unless flagged by AI).
- Each post must display:
 - Author name and profile picture
 - Timestamp (posted X minutes/hours/days ago)
 - Number of replies
 - Tags
 - View count

#### 4.2.2 Viewing a Post

- Any user (including guests) can click on a post to read its full content and all replies.
- View count must increment each time the post detail page is loaded.

#### 4.2.3 Post Feed / Listing

- The homepage must display a paginated list of all posts.
- Posts should be sorted by:
 - **Latest** (default)
 - **Most Replied**
 - **Most Viewed**
- Filtering by **tags** must also be supported.

---

### 4.3 Reply / Answer System

- A **logged-in user** can reply to any post.
- Reply fields:
 - **Content** ŌĆö The reply text (supports rich text or markdown)
 - **Screenshot / Image** ŌĆö Optional (same rules as posts: max 3 images, 5 MB each)
- Replies are displayed in chronological order under the post.
- Each reply must show:
 - Author name and profile picture
 - Timestamp
- A reply to a reply (nested comment) is optional but encouraged.

---

### 4.4 Search Functionality

- A **search bar** must be visible on all pages.
- Search must work for both **guests and logged-in users**.
- Search should cover:
 - Post titles
 - Post descriptions
 - Tags
- Search results must:
 - Be displayed as a paginated list
 - Show matched keyword highlights (optional but encouraged)
 - Show relevant posts sorted by relevance or date

> **Tip:** Implement full-text search using your database's built-in features (MongoDB Text Index, PostgreSQL `tsvector`, MySQL FULLTEXT, etc.) or a search engine like Elasticsearch / MeiliSearch.

### 4.5 AI-Powered Spam & Abuse Detection

> This is the most critical and differentiating feature of this project.

#### 4.5.1 AI Integration Requirement

- Integrate any **AI/LLM API** of your choice:
 - OpenAI GPT (gpt-3.5-turbo or gpt-4)
 - Google Gemini
 - Groq (free tier available)
 - HuggingFace Inference API
 - Ollama (for local LLM)
 - Cohere
- The AI must **analyze every new post and reply** for the following:
 - Hate speech
 - Profanity or abusive language
 - Spam content (repeated links, irrelevant promotions)
 - Harassment or personal attacks

#### 4.5.2 How Abuse Detection Works

```
User submits post/reply

Content sent to AI for analysis

 ABUSIVE? 

 YES 
 1. Flag the content as "spam/abusive" 
 2. Log the incident in database 
 3. Increment user's abuse_count by 1 
 4. Send notification to user (see 4.7) 
 5. If abuse_count == 3: 
 ŌåÆ Block the user account immediately 
 ŌåÆ Notify admin 

 NO 
 ŌåÆ Publish the post/reply normally 

```

#### 4.5.3 Abuse Count & Blocking Rules

| Abuse Count | Action |
|---|---|
| 1st offense | Content flagged + Notification sent to user |
| 2nd offense | Content flagged + Warning notification sent to user |
| 3rd offense | Content flagged + User account **blocked permanently** + Admin notified |

- A **blocked user** cannot:
 - Log in to the platform
 - Post or reply
- Blocked users see an appropriate message when they attempt to log in.
- Admin can **unblock** a user from the Admin Panel.

#### 4.5.4 AI Scanning Prompt Design

Your AI prompt should clearly instruct the model. Example prompt structure:

```
You are a content moderation assistant. Analyze the following user-submitted content 
and determine if it contains any of the following: hate speech, profanity, abusive language, 
spam, or harassment. 

Respond ONLY with a JSON object in this format:
{
 "is_abusive": true or false,
 "reason": "brief reason if abusive, else null",
 "confidence": 0.0 to 1.0
}

Content to analyze:
"[USER CONTENT HERE]"
```

> **Important:** AI scanning must happen **server-side** (not on the frontend). Never expose your AI API key to the browser.

### 4.6 Rate Limiting

Rate limiting prevents users from flooding the platform with posts or replies.

#### 4.6.1 Post & Reply Rate Limit

- A logged-in user can submit a maximum of **5 posts or replies within any 1-minute window**.
- This 5-action limit is **combined** (not 5 posts + 5 replies separately).
- On exceeding the limit, the user receives:
 > _"You are posting too fast. Please wait before submitting again."_
- The counter resets after 1 minute from the first action.
- Track rate limits using:
 - Redis (recommended)
 - In-memory store (acceptable for small scale)
 - Database timestamp-based checks (acceptable fallback)

#### 4.6.2 Login Rate Limit

_(Documented in Section 4.1.3 above)_

#### 4.6.3 Rate Limit Summary Table

| Action | Limit | Window | Block Duration |
|---|---|---|---|
| Post / Reply | 5 actions | 1 minute | Until window resets |
| Login attempts | 10 attempts | 5 minutes | 5 minutes |

### 4.7 Notification System

Users must receive **in-app notifications** (and optionally email notifications) for the following events:

| Event | Who Gets Notified |
|---|---|
| Someone replies to your post | Post author |
| Your post/reply is flagged as abusive (1st offense) | The offending user |
| Your post/reply is flagged as abusive (2nd offense) | The offending user |
| Your account is blocked due to repeated abuse (3rd offense) | The offending user + Admin |
| Admin unbans your account | The user |

#### 4.7.1 In-App Notification Requirements

- A **notification bell icon** in the navbar showing unread count.
- Clicking the bell opens a **notification dropdown or page** with all recent notifications.
- Notifications must show:
 - Notification message
 - Timestamp
 - Read/Unread status
- Mark as Read functionality must be provided.

#### 4.7.2 Email Notification (Optional but Encouraged)

- Send email for critical events: account blocked, 1st/2nd abuse warning.
- Use services like: Nodemailer, SendGrid, Resend, Mailgun.

### 4.8 Admin Panel

The Admin Panel is a **separate protected area** accessible only to users with the `admin` role.

#### 4.8.1 Admin Dashboard (Overview)

The main dashboard must display the following **platform statistics**:

- Total number of registered users
- Total number of posts
- Total number of replies
- Total number of flagged / abusive posts
- Total number of blocked users
- Posts created today / this week / this month (graph or counter)
- Active users (logged in within last 7 days)

#### 4.8.2 User Management

Admin must be able to:

- View a **paginated list of all users** with the following columns:
 - Username, Email, Registration Date, Status (Active / Blocked), Abuse Count
- **Search users** by username or email
- **Filter users** by status (Active / Blocked / All)
- **Block a user** manually (even if abuse count < 3)
- **Unblock a user**
- **View user profile** including all their posts, replies, and abuse history

#### 4.8.3 Spam / Abuse Management

Admin must be able to:

- View all **flagged / abusive posts and replies** with:
 - Content
 - Author
 - Date
 - AI confidence score
 - AI reason for flagging
- **Remove/Delete** any flagged content
- **Dismiss a flag** if AI was incorrect (false positive)
- View all users with **abuse_count > 0** in a dedicated "Spam Users" section

#### 4.8.4 Content Management

Admin must be able to:

- View **all posts** (with search and filter by tag, date, status)
- **Delete any post** or **reply** if it violates platform rules
- View **all replies** for a specific post

#### 4.8.5 Platform Performance Metrics

Admin must see a **Performance / Analytics** section showing:

- **Posts over time** ŌĆö Line or bar chart (daily/weekly/monthly)
- **New user registrations over time** ŌĆö Line chart
- **Abuse incidents over time** ŌĆö Chart showing how many posts were flagged per day/week
- **Top tags** ŌĆö Most used tags on the platform
- **Most active users** ŌĆö Users with the most posts + replies

> **Tip:** Use chart libraries like Chart.js, Recharts, ApexCharts, or any equivalent.

## 5. Non-Functional Requirements

### 5.1 Performance

- API response time should be under **500ms** for standard operations.
- Images must be compressed or resized before storage (use Cloudinary transformations or Sharp.js).
- Implement **pagination** on all list pages (posts, users, notifications).

### 5.2 Scalability

- Database schemas/models must be designed to support growth.
- Avoid N+1 query problems ŌĆö use joins, aggregations, or population efficiently.

### 5.3 Security

- **Never store plain-text passwords** ŌĆö always use bcrypt or Argon2.
- **Sanitize all user inputs** to prevent XSS and SQL/NoSQL injection.
- **API keys** (AI, email, storage) must be stored in environment variables, never hardcoded.
- All sensitive API routes must require **valid authentication tokens**.
- Implement **CORS** properly.

### 5.4 Code Quality

- Code must be well-structured and follow consistent naming conventions.
- Each major feature should be in its own module/file.
- Include a `README.md` with setup and run instructions.

### 5.5 Error Handling

- All API endpoints must return proper HTTP status codes:
 - `200` OK, `201` Created, `400` Bad Request, `401` Unauthorized, `403` Forbidden, `404` Not Found, `429` Too Many Requests, `500` Internal Server Error
- User-friendly error messages must be displayed on the frontend.

## 8. Security Requirements

| Requirement | Details |
|---|---|
| Password Hashing | Use `bcrypt` (min rounds: 10) or `argon2` |
| JWT Security | Sign with a strong secret, set expiry time |
| Input Validation | Validate and sanitize all user inputs server-side |
| XSS Prevention | Escape HTML in rendered user content |
| SQL/NoSQL Injection | Use parameterized queries or ODM (Mongoose, Prisma, etc.) |
| API Key Security | Store all keys in `.env` file ŌĆö never in code or frontend |
| CORS | Allow only trusted origins |
| HTTPS | Must be used in production |
| Admin Route Protection | All `/api/admin/*` routes require admin role middleware |
